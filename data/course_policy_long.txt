# AI Agent Engineering 수업 운영 정책 안내

이 문서는 Chapter 7 문서 청킹 실습을 위한 긴 샘플 문서입니다. Chapter 6에서는 짧은 수업 안내 문서를 문단 단위로 Chroma에 저장했습니다. Chapter 7에서는 같은 Chroma 기반 RAG 구조를 유지하면서, 긴 문서를 chunk 단위로 나누어 저장하고 검색 품질을 비교합니다. 이 문서에는 수업 개요, 실습 환경, 과제 제출, 보안, 질문 방법, 미니 프로젝트, 평가 기준 등 여러 주제가 섞여 있습니다.

## 1. 수업 개요

AI Agent Engineering 수업은 생성형 AI와 에이전트 기반 애플리케이션을 실습 중심으로 학습하는 과정입니다. 수강생은 Gemini API, 임베딩, Chroma 벡터 DB, RAG, 프롬프트 설계, 간단한 에이전트 워크플로를 단계적으로 실습합니다. 이 수업의 목표는 단순히 개념을 암기하는 것이 아니라, 실제 동작하는 작은 AI 서비스를 직접 만들어 보는 것입니다.

수업은 매주 이론 설명, 코드 실습, 실습 결과 확인, 미니 과제 순서로 진행됩니다. 실습 코드는 모두 GitHub 저장소에 정리합니다. 수강생은 각 장에서 제공되는 기본 예제를 먼저 실행한 뒤, 본인의 환경에서 결과가 어떻게 달라지는지 확인해야 합니다. 실행 결과가 다르게 나오면 에러 메시지와 실행 명령어를 함께 기록합니다.

## 2. 실습 저장소 운영 기준

실습 저장소 이름은 ai-agent-practice입니다. 모든 실습 파일은 이 저장소 안에서 관리합니다. Chapter별로 완전히 분리된 새 프로젝트를 계속 만들지 않고, 기존 구조를 유지하면서 필요한 파일을 하나씩 추가합니다. 이렇게 하면 앞 장에서 만든 Gemini 클라이언트, 임베딩 클라이언트, Chroma 클라이언트, 문서 프롬프트 함수를 다음 장에서도 재사용할 수 있습니다.

기본 폴더 구조는 app, data, outputs, docs, notebooks, solutions로 구성합니다. app 폴더에는 실행 코드와 공통 함수 파일을 둡니다. data 폴더에는 실습에 사용할 입력 문서를 둡니다. outputs 폴더에는 실행 로그, 검색 결과 비교표, 분석 결과를 저장합니다. solutions 폴더에는 강사용 또는 정답 예시 코드를 둡니다.

## 3. 실습 환경 준비

실습을 시작하기 전에 Python 가상환경을 만들고 필요한 패키지를 설치합니다. requirements.txt에는 google-genai, python-dotenv, chromadb가 포함되어 있습니다. Chapter 7의 기본 청킹 실습은 직접 구현한 character 기반 chunker를 사용하므로 추가 패키지는 필요하지 않습니다. LangChain Text Splitter는 확장 학습에서 참고할 수 있지만 기본 실습에는 사용하지 않습니다.

가상환경 폴더인 .venv는 Git에 올리지 않습니다. Python 실행 중 자동 생성되는 __pycache__ 폴더와 *.pyc 파일도 Git에 올리지 않습니다. Chroma 로컬 DB가 저장되는 chroma_db 폴더도 실행 환경마다 달라질 수 있으므로 Git에 올리지 않습니다. 반면 outputs 폴더는 실습 결과를 기록하는 용도이므로 전체를 제외하지 않습니다.

## 4. API Key 보안 정책

Gemini API Key는 반드시 .env 파일에 저장합니다. API Key를 Python 코드에 직접 작성하지 않습니다. .env 파일은 .gitignore에 포함되어 있어야 하며 GitHub에 커밋하지 않습니다. 공개 저장소에 실제 API Key가 올라가면 다른 사람이 해당 키를 사용할 수 있으므로 비용과 보안 문제가 발생할 수 있습니다.

.env.example 파일에는 실제 키가 아니라 예시 값만 작성합니다. 예를 들어 GEMINI_API_KEY=your_api_key_here와 같이 더미 값을 넣습니다. 수강생은 로컬 환경에서 .env.example을 참고하여 .env 파일을 직접 만들고, 본인의 실제 API Key를 입력합니다. 과제 제출 시에도 .env 파일은 절대 포함하지 않습니다.

## 5. 과제 제출 정책

과제 제출 기한은 매주 금요일 23:59입니다. 특별한 공지가 없는 한 지각 제출은 감점될 수 있습니다. 제출 파일에는 실행 코드, 간단한 실행 방법, 결과 요약이 포함되어야 합니다. 코드만 제출하고 실행 방법이나 결과 설명이 없으면 채점자가 의도를 파악하기 어려우므로 감점될 수 있습니다.

과제 파일명은 장 번호와 이름을 알 수 있도록 작성합니다. 예를 들어 chapter07_chunking_result.md 또는 chapter07_chunking_solution.py와 같이 작성할 수 있습니다. 노트북 파일을 제출하는 경우에는 실행 결과가 남아 있는 상태로 제출합니다. Python 파일을 제출하는 경우에는 README 또는 주석으로 실행 방법을 함께 설명합니다.

## 6. GitHub 제출 기준

실습 코드는 GitHub에 커밋하고 푸시합니다. 커밋 메시지는 작업 내용을 알 수 있도록 작성합니다. 예를 들어 Add chapter07 chunking practice structure, Implement chunk search comparison, Update chapter07 result log와 같이 작성할 수 있습니다. 의미 없는 커밋 메시지나 너무 큰 단위의 커밋은 피하는 것이 좋습니다.

커밋 전에 반드시 git status를 확인합니다. .env, chroma_db, .venv, __pycache__ 같은 파일이 포함되어 있으면 커밋하지 않습니다. 실습 결과로 생성된 outputs의 작은 Markdown 로그 파일은 커밋할 수 있습니다. 다만 대용량 임시 파일이나 개인 정보가 포함된 결과 파일은 커밋하지 않습니다.

## 7. 오류 질문 방법

오류가 발생했을 때는 단순히 안 됩니다라고 질문하지 않습니다. 질문할 때는 실행한 명령어, 전체 에러 메시지, 현재 작업 폴더, Python 버전, 설치한 패키지 목록, 시도한 해결 방법을 함께 공유합니다. 가능하면 터미널 화면 캡처나 로그 파일도 함께 첨부합니다. 이렇게 해야 조교나 튜터가 원인을 빠르게 파악할 수 있습니다.

좋은 오류 질문 예시는 다음과 같습니다. python app/chunk_index_documents.py를 실행했는데 GEMINI_API_KEY가 설정되어 있지 않다는 오류가 발생했습니다. .env 파일은 프로젝트 루트에 만들었고, 파일 내용은 GEMINI_API_KEY=... 형식입니다. python-dotenv는 설치되어 있습니다. 현재 작업 폴더는 ai-agent-practice입니다. 이런 식으로 질문하면 문제 해결이 훨씬 쉬워집니다.

## 8. Chapter 7 청킹 실습 목표

Chapter 7의 목표는 새로운 RAG 시스템을 처음부터 다시 만드는 것이 아닙니다. Chapter 6에서 만든 Chroma 기반 RAG 구조를 유지한 상태에서 문서를 나누는 방식만 개선합니다. Chapter 6에서는 빈 줄 기준으로 문단을 나누어 저장했습니다. Chapter 7에서는 긴 문서를 chunk_size와 chunk_overlap 설정에 따라 여러 chunk로 나누어 저장합니다.

핵심 변화는 문단 단위 저장에서 chunk 단위 저장으로 바뀌는 것입니다. 같은 문서라도 chunk 크기가 너무 크면 여러 주제가 섞여 검색 결과가 불필요하게 길어질 수 있습니다. 반대로 chunk 크기가 너무 작으면 질문에 답할 충분한 맥락이 사라질 수 있습니다. 따라서 chunk_size와 chunk_overlap 값을 바꾸어 검색 결과를 비교해야 합니다.

## 9. chunk size와 chunk overlap 기준

chunk size는 하나의 chunk가 가질 수 있는 최대 길이를 의미합니다. 이 실습에서는 초보자 이해를 위해 문자 수 기준으로 chunk size를 설정합니다. 예를 들어 chunk_size를 300으로 설정하면 하나의 chunk를 대략 300자 단위로 나눕니다. 실제 서비스에서는 토큰 수 기준으로 계산하는 경우가 많지만, 원리를 이해하기 위해 문자 수 기준으로 시작합니다.

chunk overlap은 인접한 chunk 사이에 일부 내용을 겹치게 하는 설정입니다. 예를 들어 chunk_size가 300이고 chunk_overlap이 50이면, 다음 chunk는 앞 chunk의 마지막 50자 정도를 포함하여 시작합니다. overlap이 있으면 중요한 문장이 chunk 경계에서 잘리는 문제를 줄일 수 있습니다. 하지만 overlap이 너무 크면 중복 저장이 많아지고 임베딩 비용과 저장 공간이 증가합니다.

## 10. metadata 설계 기준

각 chunk에는 본문뿐 아니라 metadata도 함께 저장합니다. 기본 metadata에는 source, section, paragraph_id, chunk_index, chunk_size, chunk_overlap을 포함합니다. source는 원본 파일명입니다. section은 문서의 섹션 제목입니다. paragraph_id는 원래 문단 번호이고, chunk_index는 해당 문단 안에서 몇 번째 chunk인지 나타냅니다.

metadata는 검색 결과를 해석할 때 중요합니다. 검색된 chunk가 어느 문서에서 왔는지, 어느 섹션에 있는지, 원래 문단의 어느 위치에서 만들어졌는지 알 수 있어야 합니다. RAG에서는 답변뿐 아니라 근거 제시가 중요하므로, metadata를 잘 설계하면 답변의 신뢰도를 높일 수 있습니다.

## 11. 검색 결과 비교 기준

청킹 전략을 비교할 때는 동일한 질문 세트를 사용해야 합니다. 예를 들어 API Key는 어디에 저장해야 하나요, 과제 제출 기한은 언제인가요, 오류 질문을 할 때 무엇을 함께 공유해야 하나요, 미니 프로젝트 결과물에는 무엇이 포함되어야 하나요와 같은 질문을 사용할 수 있습니다. 같은 질문으로 여러 chunk 설정을 비교해야 어떤 전략이 더 좋은지 판단할 수 있습니다.

검색 결과는 distance 값만 보고 판단하지 않습니다. 검색된 chunk가 실제로 질문에 직접 답하는지, 하나의 chunk만으로 충분한 근거가 되는지, 불필요한 내용이 너무 많이 섞이지 않았는지, source와 section metadata가 명확한지 함께 확인해야 합니다. 처음에는 사람이 직접 0점, 1점, 2점으로 평가하는 방식이 가장 이해하기 쉽습니다.

## 12. 미니 프로젝트 결과물

Chapter 7 미니 프로젝트 결과물에는 chunk 인덱싱 코드, chunk 검색 코드, 청킹 전략 비교 코드, 긴 샘플 문서, 검색 결과 비교 로그가 포함됩니다. 검색 결과 비교 로그는 outputs/chapter07_chunking_comparison.md에 정리합니다. 비교 로그에는 질문, chunk 설정, top 1 검색 결과, 관련성 점수, 근거 충분성 점수, 간단한 해석을 포함합니다.

미니 프로젝트를 제출할 때는 어떤 chunk_size와 chunk_overlap을 사용했는지 명시해야 합니다. 또한 가장 좋은 결과를 보인 설정이 무엇인지 짧게 설명해야 합니다. 정답은 하나로 고정되어 있지 않습니다. 문서 유형과 질문 유형에 따라 적절한 chunking 전략이 달라질 수 있으므로, 실험 결과를 근거로 설명하는 것이 중요합니다.

## 13. 수업 참여와 복습 안내

수업 중 실습을 따라오지 못했을 경우에는 먼저 실행 로그를 확인하고, 어느 단계에서 문제가 발생했는지 정리합니다. 단순히 최종 코드만 복사하면 다음 장에서 응용하기 어렵습니다. 각 파일이 어떤 역할을 하는지, 입력과 출력이 무엇인지, Chroma에 저장되는 데이터 구조가 무엇인지 직접 확인해야 합니다.

복습할 때는 Chapter 6의 문단 단위 RAG 흐름과 Chapter 7의 chunk 단위 RAG 흐름을 비교해 보는 것이 좋습니다. 두 장 모두 Gemini 임베딩과 Chroma를 사용하지만, 검색 대상이 되는 단위가 다릅니다. Chapter 6은 문단을 저장하고, Chapter 7은 문단을 더 작은 chunk로 나누어 저장합니다. 이 차이가 검색 품질에 어떤 영향을 주는지 관찰하는 것이 Chapter 7의 핵심입니다.
